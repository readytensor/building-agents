## Locked design decisions (decisions, not just preferences)

These have been settled across planning conversations. Build on them; don't re-derive. See `README.md` for the full rationale behind each.

- **One connected coding-agent toy across the series.** Coding agent is the worked example.
- **The toy *task* forces each episode's failure mode** â€” not extra scaffolding code. Minimal code per episode; the task's natural difficulty does the work.
- **Episode arc (code additions) â€” current, post-Ep 5 repivot:**
  - Ep 1 â€” `while`-loop + one tool (`bash`) + naive stop condition. ~60â€“80 lines.
  - Ep 2 â€” add a few general primitives (`read`, `write`, `grep`), a tiny `@tool`/schema helper.
  - Ep 3 â€” **rolling-summary compaction** + **done tool (`TaskComplete`)** â€” paired around the theme of *making long-running tasks reliable*.
  - Ep 4 â€” plan step (TODO-style tool: `write_plan`) + in-the-moment reasoning step (`think`, no-op echo) + **dynamic system-prompt mechanism** (`_system_with_plan` â€” load-bearing for Ep 5). Reflection was tried and dropped.
  - **Ep 5 â€” skills system** (`list_skills` + `load_skill` + `.skills/<name>/SKILL.md` file format + dynamic skill-body injection in system prompt + `_SKILL_TOOLS_REGISTRY` for tools that only register on skill load). ~70 LOC delta on Ep 4. *Originally planned as multi-agent; repivoted during scoping.*
  - **Ep 6 â€” orchestration** (multi-agent + `delegate` + worker runtime; skills serve as worker configurations). Not yet spec'd.
- **LLM SDK: `openai` Python package against the Chat Completions API** (not the Responses API). Provider-portable via `base_url` override. No provider-specific features used. **Dev-time exception (Eps 4â€“5):** native Anthropic SDK + prompt caching used during development for cost/speed; must be translated back to openai before publishing companion code.
- **Producer-brief framing principle:** episodes teach *how to build*, not *how agents perform*. Empirical observations from runs are brief asides (~30â€“60 sec in video, an "observation" section in the spec), not headlines. The build is the deliverable. See `memory/feedback_build_spine_principle.md`.
- **Sandbox treatment: light touch.** ~60 seconds in Ep 1 when `bash` appears, showing the `SandboxContext` pattern. Full guardrails / durable execution / production ops are out of scope.
- **Companion code repo: folder-per-episode** (`episodes/01-loop/`, etc.) with each episode containing `agent.py`, `initial/` (pristine template, committed), and `sandbox/` (agent's working dir, gitignored, reset on every run). Reset is in the first 5 lines of `agent.py` â€” `shutil.copytree(initial, sandbox)` â€” so re-running is deterministic.
- **Toy codebase: `md2html`** â€” a small but properly-structured Markdown-to-HTML CLI with real module boundaries (lexer / parser / renderer / extensions / CLI) and a pytest suite. Full spec in [`spec/md2html.md`](./spec/md2html.md) â€” architecture, markdown subset, extensions, CLI, file-by-file responsibilities, Ep 1's `initial/` state, and forward-looking sketch of per-episode divergences.
- **Capstone (real-project) episode is deferred.** Decide after Ep 6. If kept, it's a looser documentary companion piece.
- **Framing principle for scope:** *Things that change the agent's shape are in scope. Things that wrap around it aren't.* Out of scope (separate series candidates): durable execution, full guardrails, production ops, prompt engineering at the token level, framework reviews, model training/RL.

## Ep 5 carryover gotchas (read before starting Ep 6 work)

1. **Dev-time SDK swap continues.** Eps 4 and 5's `agent.py` use the **native Anthropic SDK** for prompt caching during development. **Must translate back to the openai SDK shape before publishing** companion code. Ep 6's agent.py should follow the same dev-time pattern. Header docstrings in the agent files explain. Two Anthropic-specific things to translate when shipping: prompt caching (no openai equivalent for `cache_control`), and the server-side `web_search` tool (becomes a local web_search MCP or custom impl in the published code).
2. **Prompt-caching rule remains:** dynamic per-call state goes in the **system prompt** (rebuilt as `[stable base text with cache_control, dynamic addendum without]`), NOT in messages. Ep 5 extends this to skill bodies; Ep 6's worker runtime should follow the same rule for whatever state it needs to expose per-call.
3. **`done()` reliability is mostly self-resolved.** Ep 5 runs called `done()` in 4 of 4 task-completing runs (v1, v3b, v3c, isolation), and verification skill was loaded in only 1 of those â€” so done()-reliability is **decoupled from any specific tool/skill**. Appears task-dependent: when the agent has a clean completion signal (passing tests + ticked-off plan), done() fires reliably. For Ep 6: the orchestrator-owns-done() architectural answer is still the right structural pattern, but framed as the right abstraction, not as a fix for a broken thing.
4. **Compaction-server-tool pair-splitting bug (dev-only).** When the model calls Anthropic's server-side `web_search`, `server_tool_use` + `web_search_tool_result` blocks are paired in the same assistant response. Compaction's slice points are at message boundaries; it can split these pairs across the cut, causing the next API call to reject with `unexpected tool_use_id`. **Workaround:** `EP3_THRESHOLD=200000` (skip compaction) for short trajectories. **Proper fix would walk the tail-cut boundary backward.** Not fixed because server tools disappear in the openai-SDK published code. If Ep 6 uses compaction with server tools, this needs to be fixed first.
5. **The skills purity discipline (load-bearing for Ep 6's worker runtime).** Skills in Ep 5 are stateless beyond `LOADED_SKILLS` + tool registrations. That means a fresh child agent spawned by `delegate` can call `load_skill("research")` on its first turn and be fully configured â€” no parent-to-child skill transplantation needed. Ep 6 inherits this property; don't break it.
6. **Producer-brief framing went through 3 readings before landing.** Initial spec headlines drifted into "model behavior commentary" framing ("planning costs more!", "skill loaded but never used!") rather than "here's how to build" framing. User pushed back: the series teaches *how to build agents*, not *how agents perform*. See `memory/feedback_build_spine_principle.md` â€” apply the same rule to Ep 6's producer brief. The build is the deliverable; empirical observations are brief asides.
7. **Strong task wording observation (operational, applies to Ep 6's orchestrator).** Ep 5 showed that soft instructions + competing low-friction paths â†’ unreliable skill use; strong instructions â†’ reliable. Ep 6's orchestrator constructs the task strings that get passed to children via `delegate(task, ...)`. Orchestrators that write strong, low-competing-path subtasks will get more reliable worker behavior. A design consideration for the Ep 6 build, not a lesson to dwell on.
8. **Task-choice lesson from Ep 5:** the GitHub Alerts task worked well because (a) it touched real markdown pipeline seams, (b) it had an external info source (GitHub docs) the agent had reason to consult, and (c) the fixture-pre-exists pattern matched Eps 2â€“4 convention. For Ep 6: same task is reusable (orchestrator decomposes it: research â†’ implement â†’ verify), OR pick a more parallelizable task (e.g., implement two independent markdown features at once) if the goal is to make orchestration's value more plausible *a priori*. Open decision.
